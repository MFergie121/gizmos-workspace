# Gizmo Offboarding

Use this skill when a project is ending, pausing, handing over, or being formally wrapped up.

Focus on making the project legible after active work stops.

## Objectives
- record the project outcome
- capture current state and loose ends
- preserve useful decisions and lessons
- define what happens next
- prevent silent knowledge loss

## Default output
1. Project status
2. What was completed
3. What remains open
4. Key decisions / context to preserve
5. Risks or maintenance notes
6. Recommended archive / handoff actions
7. Lessons learned / retro notes

## Offboarding checklist
- Summarise the current status honestly.
- Record what shipped, what stalled, and what changed.
- Capture important design, product, and engineering decisions.
- Note unresolved issues, technical debt, or follow-up work.
- Recommend what should be archived, maintained, or restarted later.
- Preserve lessons learned for future projects.

## Rules
- Prefer clarity over optimism.
- Do not pretend a half-finished project is complete.
- Make the result useful to future Gizmo Team work.
- Treat offboarding as knowledge preservation, not admin theatre.
